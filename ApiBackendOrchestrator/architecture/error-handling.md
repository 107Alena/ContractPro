# Обработка ошибок и отказоустойчивость API/Backend Orchestrator

В рамках документа описана стратегия обработки ошибок, retry-политики, circuit breaker, timeout-политики, graceful degradation и graceful shutdown для компонента **API/Backend Orchestrator** сервиса **ContractPro**.

Документ является частью архитектурной документации оркестратора и дополняет [high-architecture.md](high-architecture.md).

---

# 1. Единый формат ошибок для frontend

## 1.1 Структура ответа об ошибке

Все HTTP-ответы об ошибках, возвращаемые оркестратором, используют единый JSON-формат. Формат обеспечивает:

- **Машиночитаемость** — `error_code` для программной обработки на frontend.
- **Человекочитаемость** — `message` на русском языке для отображения пользователю (NFR-5.2).
- **Подсказку следующего действия** — `suggestion` объясняет пользователю, что делать дальше.
- **Трассируемость** — `correlation_id` для обращения в поддержку и поиска в логах.

```json
{
  "error_code": "FILE_TOO_LARGE",
  "message": "Файл превышает максимальный размер 20 МБ",
  "details": {
    "max_size_bytes": 20971520,
    "actual_size_bytes": 31457280
  },
  "correlation_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "suggestion": "Попробуйте загрузить файл меньшего размера"
}
```

### Описание полей

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| `error_code` | `string` | Обязательно | Машиночитаемый код ошибки. Формат: `UPPER_SNAKE_CASE`. Полный каталог — раздел 3. |
| `message` | `string` | Обязательно | Сообщение на русском языке для отображения пользователю. Не содержит технических деталей (stack trace, имена сервисов). |
| `details` | `object \| null` | Опционально | Дополнительный контекст. Структура зависит от `error_code`. Для `VALIDATION_ERROR` — массив ошибок валидации. Для `FILE_TOO_LARGE` — лимиты. Для 5xx — `null` (внутренние детали не раскрываются клиенту). |
| `correlation_id` | `string` | Обязательно | UUID v4. Генерируется оркестратором при приёме запроса. Совпадает с `correlation_id` в логах и downstream-вызовах. Пользователь может указать этот ID при обращении в поддержку. |
| `suggestion` | `string \| null` | Опционально | Рекомендация следующего действия на русском языке. Для 4xx — конкретная подсказка. Для 5xx — общая рекомендация «Попробуйте позже». Для 401/403 — рекомендация по авторизации. |

## 1.2 Контракт с frontend

Frontend ДОЛЖЕН:

1. Проверять наличие `error_code` для программной обработки (не парсить `message`).
2. Отображать `message` пользователю как есть (текст на русском языке).
3. Отображать `suggestion`, если он не `null`, как дополнительную подсказку.
4. Сохранять `correlation_id` для отображения пользователю при ошибках 5xx (кнопка «Скопировать ID ошибки»).
5. Использовать `details` для обогащения UI (например, подсветка невалидных полей при `VALIDATION_ERROR`).

## 1.3 Принципы формирования сообщений

- **Язык:** Русский (NFR-5.2).
- **Тон:** Нейтральный, без технического жаргона.
- **Формат:** Полное предложение с точкой в конце.
- **Безопасность:** Сообщения для 5xx НЕ раскрывают внутреннюю архитектуру, имена сервисов, stack trace.
- **Консистентность:** Один `error_code` всегда сопровождается одним шаблоном `message` (параметризованным при необходимости).

# 2. Маппинг HTTP-статус-кодов

## 2.1 Таблица маппинга

| HTTP код | Категория | Когда возвращается | Пример |
|----------|-----------|-------------------|--------|
| **400** | Невалидный ввод | Нарушение формата запроса, отсутствие обязательных полей, невалидный UUID, некорректные параметры пагинации | Отсутствует поле `title` при загрузке, `page=-1` |
| **401** | Ошибка аутентификации | Отсутствует JWT, истёкший токен, невалидная подпись, повреждённый токен | `Authorization` header отсутствует или содержит невалидный Bearer token |
| **403** | Недостаточно прав | Роль пользователя не позволяет выполнить операцию | `BUSINESS_USER` пытается получить полные результаты анализа или зайти в admin-раздел |
| **404** | Ресурс не найден | Документ, версия, артефакт или diff не существует; документ принадлежит другой организации (tenant isolation — возвращаем 404, а не 403, чтобы не раскрывать существование ресурса) | `GET /contracts/{id}` с несуществующим ID |
| **409** | Конфликт состояния | Операция невозможна в текущем состоянии ресурса | Архивация уже архивированного документа, повторная проверка версии в статусе `PENDING` |
| **413** | Файл слишком большой | Загружаемый файл превышает лимит 20 МБ | `POST /contracts/upload` с файлом 25 МБ |
| **415** | Неподдерживаемый формат | Загружаемый файл не является PDF (проверка по MIME-типу и magic bytes) | Загрузка `.docx` или `.jpg` |
| **429** | Превышен лимит запросов | Per-organization rate limit исчерпан | Организация превысила лимит загрузок или запросов в единицу времени |
| **500** | Внутренняя ошибка | Непредвиденная ошибка в оркестраторе (паника, ошибка маршализации, некорректная конфигурация) | Неожиданный `nil` pointer, ошибка сериализации |
| **502** | Ошибка downstream-сервиса | DM, OPM, Object Storage вернули ошибку или недоступны после всех retry | DM вернул 500, Object Storage не отвечает |
| **503** | Сервис недоступен | Оркестратор не готов к обработке запросов (readiness probe failed) | При старте до установления соединений с Redis, RabbitMQ, DM |

## 2.2 Дополнительные HTTP-заголовки при ошибках

| HTTP код | Заголовок | Значение |
|----------|-----------|----------|
| 429 | `Retry-After` | Количество секунд до восстановления лимита (например, `5`) |
| 503 | `Retry-After` | Предполагаемое время восстановления (например, `30`) |
| Все ошибки | `X-Correlation-Id` | UUID, дублирует `correlation_id` из тела ответа |

## 2.3 Маппинг ошибок downstream-сервисов

При проксировании запросов к DM, OPM, UOM оркестратор **не пробрасывает** внутренние ошибки напрямую. Вместо этого маппинг:

| Downstream HTTP код | Действие оркестратора |
|---------------------|----------------------|
| 400 | Анализ. Если ошибка вызвана данными пользователя — 400 с адаптированным сообщением. Если внутренний рассинхрон — 500 + alert. |
| 404 | 404 с user-friendly сообщением (`DOCUMENT_NOT_FOUND`, `VERSION_NOT_FOUND`). |
| 409 | 409 с маппингом DM conflict-кода в пользовательский (`DOCUMENT_ARCHIVED`, `VERSION_STILL_PROCESSING`). |
| 5xx | Retry (по политике раздела 4). При исчерпании — 502. |
| Timeout | Retry. При исчерпании — 502. |
| Connection refused | Retry. При исчерпании — 502. Circuit breaker (раздел 5). |

## 3.8 Ошибки валидации (VALIDATION_*)

| Код ошибки | HTTP | Описание | message (рус.) | suggestion (рус.) | details |
|------------|------|----------|----------------|-------------------|---------| 
| `VALIDATION_ERROR` | 400 | Ошибка валидации входных данных (невалидные поля, отсутствие обязательных параметров) | Данные запроса содержат ошибки. | Исправьте указанные ошибки и повторите запрос. | `{fields: ValidationFieldError[]}` |

### Структура `details` для `VALIDATION_ERROR`

Гарантированный контракт для маппинга на формы (React Hook Form, etc.) — массив `details.fields`, каждый элемент:

| Поле | Тип | Обязательно | Описание |
|------|-----|-------------|----------|
| `field` | `string` | Да | Путь к полю в формате JSONPath-lite: `title`, `parties.0.name`, `policy.severity`. Совместим с `setName` React Hook Form. |
| `code` | `string` (enum) | Да | Машинный код причины. См. таблицу ниже — закрытый whitelist |
| `message` | `string` | Да | Человекочитаемое описание на русском (NFR-5.2). Frontend может проигнорировать в пользу собственной i18n по `code` |
| `params` | `object \| null` | Нет | Параметры для интерполяции в i18n: для `TOO_LONG` — `{max: 100}`, для `OUT_OF_RANGE` — `{min: 0, max: 1}` |

### Whitelist `code` для validation errors

| `code` | Семантика | Типичный `params` | Пример `message` |
|--------|-----------|-------------------|------------------|
| `REQUIRED` | Обязательное поле отсутствует или пустое | — | «Поле обязательно для заполнения» |
| `INVALID_FORMAT` | Не соответствует формату (не подходит другому коду) | `{format: "email"}` | «Некорректный формат электронной почты» |
| `INVALID_UUID` | Невалидный UUID | — | «Некорректный формат идентификатора» |
| `INVALID_DATE` | Невалидная дата (формат или невозможное значение) | — | «Некорректный формат даты» |
| `INVALID_ENUM` | Значение вне разрешённого enum | `{allowed: ["a","b"]}` | «Недопустимое значение» |
| `NOT_IN_WHITELIST` | Значение не в бизнес-whitelist (динамический список, в отличие от `INVALID_ENUM`) | `{allowed_count: 12}` | «Тип договора не поддерживается» |
| `TOO_SHORT` | Длина строки меньше минимальной | `{min: 3}` | «Минимум 3 символа» |
| `TOO_LONG` | Длина строки больше максимальной | `{max: 200}` | «Максимум 200 символов» |
| `OUT_OF_RANGE` | Числовое значение вне допустимого диапазона | `{min: 0, max: 1}` | «Значение должно быть от 0 до 1» |
| `MISMATCH` | Значение не совпадает с другим полем (например, повтор пароля) | `{other_field: "password"}` | «Пароли не совпадают» |
| `DUPLICATE` | Значение уже существует (unique constraint, например, organization name) | — | «Уже используется» |
| `INVALID_REFERENCE` | Ссылка на несуществующий ресурс | `{resource_type: "policy"}` | «Указанная политика не существует» |

> **Расширение whitelist:** новые `code` добавляются только через PR с обновлением этой таблицы. Frontend маппит unknown `code` на fallback `'INVALID_FORMAT'` + показывает `message` как есть.

| `RESULTS_NOT_READY` | 409 | Да (после завершения обработки) |
| `RATE_LIMIT_EXCEEDED` | 429 | Да (через `Retry-After` секунд) |
| `STORAGE_UNAVAILABLE` | 502 | Да (через 30–60 секунд) |
| `DM_UNAVAILABLE` | 502 | Да (через 30–60 секунд) |
| `OPM_UNAVAILABLE` | 502 | Да (через 30–60 секунд) |
| `UOM_UNAVAILABLE` | 502 | Да (через 30–60 секунд) |
| `BROKER_UNAVAILABLE` | 502 | Да (через 30–60 секунд) |
| `VALIDATION_ERROR` | 400 | Нет (нужно исправить данные) |
| `INTERNAL_ERROR` | 500 | Возможно (зависит от причины) |

# 4. Retry-стратегия для downstream-сервисов

## 4.1 Общие принципы

- **Exponential backoff** — задержка между retry увеличивается экспоненциально.
- **Jitter** — к каждой задержке добавляется случайный компонент (0–50% от базовой задержки) для предотвращения thundering herd.
- **Retryable ошибки** — только транзиентные: 5xx, timeout, connection refused/reset. Клиентские ошибки (4xx) никогда не retryable.
- **Бюджет retry** — общее время retry не должно превышать timeout операции (раздел 6).
- **Логирование** — каждая попытка retry логируется на уровне `WARN` с `correlation_id`, номером попытки, ошибкой.

## 4.2 DM Client (sync REST)

DM — основная зависимость оркестратора. Все CRUD-операции с документами, версиями и артефактами проходят через DM sync REST API.

| Параметр | Значение |
|----------|----------|
| Retryable ошибки | HTTP 5xx, timeout, connection refused, connection reset |
| Не retryable | HTTP 4xx (400, 404, 409 — клиентские ошибки) |
| Макс. попыток | 3 (1 основная + 2 retry) |
| Backoff | Exponential: 200ms, 400ms, 800ms (+ jitter) |
| Timeout на запрос (read) | 5s |
| Timeout на запрос (write) | 10s |
| Circuit breaker | Да (раздел 5) |
| При исчерпании retry | HTTP 502 (`DM_UNAVAILABLE`) клиенту |

**Пример retry-последовательности (read):**

```
Попытка 1: GET /documents/{id} → timeout (5s)
  ↓ 200ms + jitter
Попытка 2: GET /documents/{id} → 503 Service Unavailable
  ↓ 400ms + jitter
Попытка 3: GET /documents/{id} → 200 OK ✓
```

## 4.3 Object Storage Client (S3)

Object Storage используется для загрузки исходных PDF-файлов перед созданием версии в DM.

| Параметр | Значение |
|----------|----------|
| Retryable ошибки | HTTP 5xx, timeout, connection error |
| Не retryable | HTTP 4xx (403 Forbidden, 404 Not Found) |
| Макс. попыток | 3 (1 основная + 2 retry) |
| Backoff | Exponential: 500ms, 1s, 2s (+ jitter) |
| Timeout на upload | 30s |
| Timeout на прочие операции (HEAD, DELETE) | 15s |
| Circuit breaker | Да (раздел 5) |
| При исчерпании retry | HTTP 502 (`STORAGE_UNAVAILABLE`) клиенту. Cleanup не требуется (файл не был загружен). |

**Особенности S3 retry:**

- При ошибке upload после частичной загрузки — не требуется cleanup: Object Storage использует multipart upload с автоматическим abort незавершённых частей (server-side TTL).
- Backoff для S3 длиннее, чем для DM, т.к. S3-операции тяжелее (network I/O с blob-данными).

## 4.4 RabbitMQ publish

RabbitMQ используется для публикации команд в DP (`ProcessDocumentRequested`, `CompareDocumentVersionsRequested`).

| Параметр | Значение |
|----------|----------|
| Retryable ошибки | Connection error, timeout, channel closed |
| Не retryable | Ошибка сериализации (программная ошибка) |
| Макс. попыток | 3 (1 основная + 2 retry) |
| Backoff | Exponential: 100ms, 200ms, 400ms (+ jitter) |
| Timeout на publish | 10s |
| Publisher confirms | Да (ожидание подтверждения от брокера) |
| Circuit breaker | Нет (используется native reconnect RabbitMQ-клиента) |
| При исчерпании retry | Логирование `CRITICAL`. Версия уже создана в DM со статусом `PENDING`. Команда может быть переотправлена оператором. Alert. |

**Важно:** При неудаче publish команды DP — пользователь получает HTTP 202 Accepted (версия создана в DM), но обработка не начнётся. Оркестратор:

1. Логирует `CRITICAL` с полным контекстом (`correlation_id`, `job_id`, `document_id`, `version_id`).
2. Генерирует alert в monitoring-систему.
3. Записывает сообщение в Redis (`failed_commands:{correlation_id}`) для последующего replay.

## 4.5 OPM Client (sync REST)

OPM используется для проксирования административных запросов (R-3: управление политиками).

| Параметр | Значение |
|----------|----------|
| Retryable ошибки | HTTP 5xx, timeout, connection refused |
| Не retryable | HTTP 4xx |
| Макс. попыток | 3 (1 основная + 2 retry) |
| Backoff | Exponential: 200ms, 400ms, 800ms (+ jitter) |
| Timeout на запрос | 5s (read), 10s (write) |
| Circuit breaker | Нет (OPM — опциональная зависимость, запросы редкие) |
| При исчерпании retry | HTTP 502 (`OPM_UNAVAILABLE`) клиенту |

## 4.6 UOM Client (sync REST — auth proxy)

UOM используется для проксирования аутентификационных запросов (`/auth/login`, `/auth/refresh`, `/auth/logout`).

| Параметр | Значение |
|----------|----------|
| Retryable ошибки | HTTP 5xx, timeout, connection refused |
| Не retryable | HTTP 4xx (401 — невалидные credentials, не retryable) |
| Макс. попыток | 2 (1 основная + 1 retry) |
| Backoff | Фиксированный: 200ms |
| Timeout на запрос | 5s |
| Circuit breaker | Нет (запросы критичны для UX, fast fail предпочтительнее) |
| При исчерпании retry | HTTP 502 клиенту. Существующие JWT продолжают валидироваться локально по публичному ключу. |

**Обоснование 2 попыток:** Для login/refresh пользователь ожидает быстрый ответ. Длительный retry недопустим — лучше быстро вернуть ошибку.

## 4.7 Сводная таблица retry-политик

| Downstream | Макс. попыток | Backoff (мс) | Timeout | При исчерпании |
|------------|--------------|--------------|---------|----------------|
| DM (read) | 3 | 200, 400, 800 | 5s | 502 |
| DM (write) | 3 | 200, 400, 800 | 10s | 502 |
| Object Storage (upload) | 3 | 500, 1000, 2000 | 30s | 502 |
| Object Storage (прочее) | 3 | 500, 1000, 2000 | 15s | 502 |
| RabbitMQ publish | 3 | 100, 200, 400 | 10s | CRITICAL log + alert |
| OPM (read) | 3 | 200, 400, 800 | 5s | 502 |
| OPM (write) | 3 | 200, 400, 800 | 10s | 502 |
| UOM | 2 | 200 | 5s | 502 |

# 5. Circuit breaker

## 5.1 Назначение

Circuit breaker предотвращает каскадный отказ при длительной недоступности downstream-сервиса. Вместо того чтобы тратить ресурсы на заведомо неуспешные запросы (с retry и timeout), circuit breaker мгновенно возвращает ошибку (**fast fail**).

**Библиотека:** `gobreaker` (совместимость с подходом DM).

## 5.2 Компоненты с circuit breaker

Circuit breaker применяется к двум downstream-клиентам:

### DM Client

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Name | `dm-client` | Для идентификации в метриках |
| Failure threshold | 5 последовательных ошибок → `OPEN` | 5 ошибок за короткое время — явный признак недоступности DM |
| Open duration | 30s → `HALF-OPEN` | 30s достаточно для recovery типичных transient-ошибок |
| Half-open: test requests | 3 запроса | 3 успешных запроса подтверждают восстановление |
| Half-open → CLOSED | 3/3 успешных test request | Возврат к нормальной работе |
| Half-open → re-OPEN | Любой failed test request | DM ещё не восстановился |
| Tracked errors | Timeout, HTTP 5xx, connection refused/reset | Только инфраструктурные ошибки |
| Not tracked (passthrough) | HTTP 4xx (400, 404, 409) | Клиентские ошибки — не признак деградации DM |

### Object Storage Client

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Name | `object-storage-client` | Для идентификации в метриках |
| Failure threshold | 5 последовательных ошибок → `OPEN` | Аналогично DM |
| Open duration | 30s → `HALF-OPEN` | Аналогично DM |
| Half-open: test requests | 3 запроса | Аналогично DM |
| Tracked errors | Timeout, HTTP 5xx, connection error | Инфраструктурные ошибки S3 |
| Not tracked | HTTP 4xx | Клиентские ошибки |

## 5.3 Компоненты без circuit breaker

| Компонент | Причина отсутствия |
|-----------|-------------------|
| RabbitMQ Client | Используется native reconnect RabbitMQ-клиента с automatic recovery. Circuit breaker не нужен — клиент сам управляет переподключением. |
| OPM Client | OPM — опциональная зависимость. Запросы редкие (только admin). Отдельный circuit breaker не оправдан. |
| UOM Client | Запросы к UOM — только login/refresh (редкие). JWT валидируется локально. Fast fail (2 попытки) достаточен. |
| Redis Client | Redis — ephemeral state. При недоступности — graceful degradation (раздел 7). Circuit breaker добавит сложности без значимой выгоды. |

## 5.4 Поведение при OPEN circuit

| Circuit breaker | Состояние OPEN | Поведение оркестратора |
|-----------------|----------------|----------------------|
| `dm-client` | DM недоступен | Мгновенный HTTP 502 (`DM_UNAVAILABLE`) без попытки запроса к DM. Readiness probe → `NOT READY`. |
| `object-storage-client` | S3 недоступен | Мгновенный HTTP 502 (`STORAGE_UNAVAILABLE`) для upload-операций. Read-операции через DM не затронуты. |

## 5.5 Мониторинг circuit breaker

Метрики (Prometheus):

| Метрика | Тип | Описание |
|---------|-----|----------|
| `orch_circuit_breaker_state` | Gauge | Текущее состояние: 0=CLOSED, 1=HALF_OPEN, 2=OPEN. Labels: `name`. |
| `orch_circuit_breaker_transitions_total` | Counter | Количество переходов между состояниями. Labels: `name`, `from`, `to`. |
| `orch_circuit_breaker_rejected_total` | Counter | Количество запросов, отклонённых OPEN circuit. Labels: `name`. |

**Алерты:**

- `CircuitBreakerOpen` — circuit breaker в состоянии OPEN > 60s → `CRITICAL` alert.
- `CircuitBreakerFlapping` — > 5 переходов OPEN → CLOSED за 5 минут → `WARNING` alert.

## 5.6 Диаграмма состояний circuit breaker

```
                     5 consecutive failures
            ┌──────────────────────────────────────┐
            │                                      ▼
        ┌────────┐                           ┌──────────┐
        │ CLOSED │                           │   OPEN   │
        │        │                           │          │
        │ Normal │                           │ Fast fail│
        │ operation                          │ all reqs │
        └────────┘                           └────┬─────┘
            ▲                                     │
            │                              30s timer
            │                                     │
            │         3/3 success                 ▼
            │◄───────────────────────── ┌───────────┐
            │                           │ HALF-OPEN │
            │                           │           │
            └───────────────────────── │ 3 test    │
              any failure → re-OPEN    │ requests  │
                                       └───────────┘
```

# 6. Timeout-политики

## 6.1 Таблица timeout-политик

| Операция | Timeout | Env-переменная | Действие при срабатывании |
|----------|---------|----------------|--------------------------|
| **HTTP-запрос (read endpoints)** | 30s | `ORCH_HTTP_READ_TIMEOUT` | HTTP 504 Gateway Timeout. Логирование `ERROR`. |
| **HTTP-запрос (upload endpoint)** | 60s | `ORCH_HTTP_UPLOAD_TIMEOUT` | HTTP 504 Gateway Timeout. Cleanup загруженных частей. Логирование `ERROR`. |
| **DM Client (read)** | 5s | `ORCH_DM_READ_TIMEOUT` | Retry (раздел 4.2). При исчерпании — HTTP 502 (`DM_UNAVAILABLE`). |
| **DM Client (write)** | 10s | `ORCH_DM_WRITE_TIMEOUT` | Retry. При исчерпании — HTTP 502 (`DM_UNAVAILABLE`). Cleanup (если файл уже загружен в S3). |
| **Object Storage upload** | 30s | `ORCH_S3_UPLOAD_TIMEOUT` | Retry (раздел 4.3). При исчерпании — HTTP 502 (`STORAGE_UNAVAILABLE`). |
| **Object Storage (HEAD, DELETE)** | 15s | `ORCH_S3_OP_TIMEOUT` | Retry. При исчерпании — логирование `ERROR` (для DELETE — orphaned object alert). |
| **RabbitMQ publish** | 10s | `ORCH_BROKER_PUBLISH_TIMEOUT` | Retry (раздел 4.4). При исчерпании — `CRITICAL` log + alert. |
| **Redis операция** | 2s | `ORCH_REDIS_TIMEOUT` | Fallback (раздел 7): пропуск rate limit / SSE degradation. Логирование `WARN`. |
| **OPM Client** | 5s (read), 10s (write) | `ORCH_OPM_READ_TIMEOUT`, `ORCH_OPM_WRITE_TIMEOUT` | Retry. При исчерпании — HTTP 502 (`OPM_UNAVAILABLE`). |
| **UOM Client** | 5s | `ORCH_UOM_TIMEOUT` | Retry (1 раз). При исчерпании — HTTP 502. |
| **SSE heartbeat interval** | 15s | `ORCH_SSE_HEARTBEAT_INTERVAL` | Отправка `:ping\n\n` keepalive каждые 15s. Если клиент не отвечает — cleanup соединения. |
| **SSE connection max age** | 24h | `ORCH_SSE_MAX_AGE` | Принудительное закрытие SSE-соединения. Клиент автоматически переподключается (EventSource API). |
| **Graceful shutdown** | 30s | `ORCH_SHUTDOWN_TIMEOUT` | Force kill процесса (раздел 9). |

## 6.2 Иерархия timeout

Timeout верхнего уровня (HTTP request) является **бюджетом** для всех внутренних операций. Внутренние timeout меньше внешнего, чтобы оркестратор всегда успел сформировать ответ:

```
HTTP request timeout (30s read / 60s upload)
  └── DM Client timeout (5s read / 10s write) × retry attempts
  └── S3 timeout (30s upload) × retry attempts
  └── RabbitMQ publish timeout (10s) × retry attempts
  └── Redis timeout (2s) — fallback, не блокирует
  └── Response serialization + overhead (~100ms)
```

**Пример worst case для upload (60s бюджет):**

```
S3 upload:    30s timeout × 1 attempt  = 30s (failure)
              2s  backoff
S3 upload:    30s timeout × 2 attempt  = 30s (failure) — уже ~62s > 60s бюджет
```

В этом случае HTTP timeout (60s) сработает раньше, чем завершится второй retry S3. Оркестратор корректно обработает context cancellation и вернёт HTTP 504.

## 6.3 Context propagation

Все timeout реализуются через `context.Context` с `context.WithTimeout`:

- HTTP handler создаёт context с общим timeout (30s/60s).
- Каждый downstream-вызов создаёт дочерний context с собственным timeout.
- При отмене родительского context — все дочерние автоматически отменяются.
- `correlation_id` передаётся через context для трассируемости.

# 7. Graceful degradation

## 7.1 Принципы деградации

При частичной недоступности downstream-сервисов оркестратор следует принципу **«отдать максимум доступного, явно сообщив о деградации»** (NFR-2.5):

1. Если зависимость критическая (DM) — fast fail с понятной ошибкой.
2. Если зависимость опциональная (OPM, Redis) — продолжение работы с ограничениями.
3. Пользователь всегда получает ответ в предсказуемом формате (раздел 1).

## 7.2 Матрица деградации

### DM недоступен

| Аспект | Поведение |
|--------|-----------|
| Загрузка договора | HTTP 502 (`DM_UNAVAILABLE`). Файл не загружается в S3 (чтобы избежать orphaned objects). |
| Список договоров | HTTP 502 (`DM_UNAVAILABLE`). |
| Получение результатов | HTTP 502 (`DM_UNAVAILABLE`). |
| Экспорт отчёта | HTTP 502 (`DM_UNAVAILABLE`). |
| Повторная проверка | HTTP 502 (`DM_UNAVAILABLE`). |
| Сравнение версий | HTTP 502 (`DM_UNAVAILABLE`). |
| SSE | Продолжает работать для DP-событий. DM-события не поступают. |
| Health check `/readyz` | HTTP 503 NOT READY. Load balancer перестаёт направлять трафик. |
| Мониторинг | Alert `CircuitBreakerOpen(dm-client)` → `CRITICAL`. |

### Object Storage недоступен

| Аспект | Поведение |
|--------|-----------|
| Загрузка договора | HTTP 502 (`STORAGE_UNAVAILABLE`). |
| Список договоров | Работает (данные из DM). |
| Получение результатов | Работает (артефакты из DM, DM хранит blob отдельно). |
| Экспорт отчёта | Работает (presigned URL формируется DM). |
| Health check `/readyz` | Остаётся 200 OK (S3 не входит в readiness check — используется только для upload). |
| Мониторинг | Alert `CircuitBreakerOpen(object-storage-client)` → `CRITICAL`. |

### RabbitMQ недоступен

| Аспект | Поведение |
|--------|-----------|
| Загрузка договора | Файл загружается в S3, версия создаётся в DM. Команда DP НЕ публикуется. HTTP 202 Accepted (версия в статусе `PENDING`). Логирование `CRITICAL`. Alert. |
| SSE-события от DP/DM | Не поступают. Пользователь не видит обновления статуса. Fallback: polling endpoint. |
| Сравнение версий | HTTP 502 (`BROKER_UNAVAILABLE`). |
| Health check `/readyz` | HTTP 503 NOT READY. |
| Мониторинг | Alert `RabbitMQConnectionLost` → `CRITICAL`. |
| Восстановление | При reconnect — оператор переотправляет failed commands из Redis (`failed_commands:*`). |

### Redis недоступен

| Аспект | Поведение |
|--------|-----------|
| Rate limiting | **Отключается** (allow all). Логирование `WARN`. Риск перегрузки downstream-сервисов компенсируется их собственными rate limiters. |
| SSE | **Деградация**: события доставляются только клиентам, подключённым к тому же инстансу оркестратора, который получил RabbitMQ-событие. Кросс-инстансный broadcast через Redis Pub/Sub не работает. |
| Upload tracking | Недоступно. Повторный запрос статуса загрузки → fallback на DM API (`GET /documents/{id}/versions/{vid}`). |
| Failed command replay | Недоступен. |
| Health check `/readyz` | HTTP 503 NOT READY. |
| Мониторинг | Alert `RedisConnectionLost` → `WARNING`. |

### OPM недоступен

| Аспект | Поведение |
|--------|-----------|
| Admin-эндпоинты (политики, чек-листы) | HTTP 502 (`OPM_UNAVAILABLE`). |
| Загрузка / анализ договоров | **Не затронуты** — оркестратор не обращается к OPM при загрузке. |
| Health check `/readyz` | Остаётся 200 OK (OPM — опциональная зависимость). |
| Мониторинг | Alert `OPMUnavailable` → `WARNING`. |

### UOM недоступен

| Аспект | Поведение |
|--------|-----------|
| Login (`POST /auth/login`) | HTTP 502. Пользователь не может войти в систему. |
| Token refresh (`POST /auth/refresh`) | HTTP 502. При истечении access token — пользователь вынужден ждать восстановления UOM. |
| JWT-валидация на запросах | **Продолжает работать** — JWT валидируется локально по публичному ключу (ASSUMPTION-ORCH-02). |
| Профиль пользователя | Кэшированный профиль в Redis (если доступен). Иначе — данные из JWT claims. |
| Health check `/readyz` | Остаётся 200 OK (UOM не входит в readiness check — JWT валидируется локально). |
| Мониторинг | Alert `UOMUnavailable` → `WARNING`. |

# 8. Health check endpoints

## 8.1 Liveness probe — `GET /healthz`

**Назначение:** Подтверждение, что процесс оркестратора жив и способен обрабатывать запросы.

**Поведение:**
- Всегда возвращает HTTP 200, если процесс запущен.
- Не проверяет downstream-зависимости.
- Используется Kubernetes / Docker для определения, нужен ли restart контейнера.

**Ответ (200):**

```json
{
  "status": "alive"
}
```

## 8.2 Readiness probe — `GET /readyz`

**Назначение:** Подтверждение, что оркестратор готов принимать пользовательский трафик.

**Проверки:**

| Зависимость | Проверка | Timeout проверки | Обязательность |
|-------------|----------|-----------------|----------------|
| Redis | `PING` command | 2s | Да |
| RabbitMQ | Connection state (connected + channels open) | 2s | Да |
| DM | `GET /healthz` (HTTP 200) | 5s | Да |

**Логика:**
- Все три проверки выполняются **параллельно**.
- Если **все** проверки пройдены — HTTP 200.
- Если **хотя бы одна** проверка не пройдена — HTTP 503.

**Ответ (200 — all healthy):**

```json
{
  "status": "ready",
  "checks": {
    "redis": {"status": "up", "latency_ms": 1},
    "rabbitmq": {"status": "up", "latency_ms": 3},
    "dm": {"status": "up", "latency_ms": 12}
  }
}
```

**Ответ (503 — degraded):**

```json
{
  "status": "not_ready",
  "checks": {
    "redis": {"status": "up", "latency_ms": 1},
    "rabbitmq": {"status": "up", "latency_ms": 3},
    "dm": {"status": "down", "error": "connection refused", "latency_ms": 5000}
  }
}
```

## 8.3 Эндпоинты без аутентификации

Следующие эндпоинты доступны без JWT (исключены из JWT Auth Middleware):

| Эндпоинт | Назначение |
|----------|-----------|
| `GET /healthz` | Liveness probe |
| `GET /readyz` | Readiness probe |
| `GET /metrics` | Prometheus metrics |
| `POST /api/v1/auth/login` | Аутентификация |
| `POST /api/v1/auth/refresh` | Обновление токена |

## 8.4 Частота опроса

| Probe | Рекомендуемый интервал | Failure threshold |
|-------|----------------------|-------------------|
| Liveness (`/healthz`) | 10s | 3 последовательных неудачи → restart |
| Readiness (`/readyz`) | 5s | 1 неудача → stop traffic |

# 9. Graceful shutdown

## 9.1 Назначение

Graceful shutdown обеспечивает корректное завершение работы оркестратора без потери данных и обрыва соединений. Это критически важно при rolling deployment, масштабировании и обновлении.

## 9.2 Последовательность shutdown

Graceful shutdown выполняется в строгом порядке. Общий бюджет времени: `ORCH_SHUTDOWN_TIMEOUT` (по умолчанию 30s).

```
Сигнал SIGTERM / SIGINT
  │
  ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 1: Mark NOT READY (0–5s)                            │
│   Readiness probe /readyz → HTTP 503                     │
│   Load balancer перестаёт направлять новый трафик         │
│   Задержка 3–5s для propagation в LB / Kubernetes        │
└──────────────────────────────────┬───────────────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 2: Stop accepting new connections                    │
│   HTTP listener перестаёт принимать новые TCP-соединения  │
└──────────────────────────────────┬───────────────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 3: Wait for in-flight HTTP requests (max 10s)        │
│   Ожидание завершения текущих HTTP-запросов               │
│   context.WithTimeout(10s) на drain                      │
│   Если не завершились за 10s — принудительное закрытие    │

                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 4: Close SSE connections                             │
│   Отправка SSE close event: event: close\ndata: {}\n\n   │
│   Клиенты получают close event и знают, что нужно        │
│   переподключиться к другому инстансу                    │
│   Удаление регистрации из Redis SSE Connection Registry  │
└──────────────────────────────────┬───────────────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 5: Stop RabbitMQ consumers                           │
│   consumer.Cancel() — прекращение получения новых         │
│   сообщений из очереди                                   │
│   Ожидание завершения in-flight message обработки        │
│   ACK для всех обработанных сообщений                    │
│   NACK + requeue для необработанных                      │
└──────────────────────────────────┬───────────────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 6: Close Redis connections                           │
│   Отписка от Redis Pub/Sub channels                      │
│   Закрытие Redis connection pool                         │
└──────────────────────────────────┬───────────────────────┘
                                   │
                                   ▼
┌──────────────────────────────────────────────────────────┐
│ Шаг 7: Close HTTP server                                 │
│   http.Server.Shutdown() — финализация                   │
└──────────────────────────────────┬───────────────────────┘
                                   │

┌──────────────────────────────────────────────────────────┐
```

## 9.3 Бюджет времени

| Шаг | Описание | Макс. время | Накопленное время |
|-----|----------|-------------|-------------------|
| 1 | Mark NOT READY + LB propagation | 5s | 5s |
| 2 | Stop accepting connections | ~0s | 5s |
| 3 | Drain in-flight HTTP requests | 10s | 15s |
| 4 | Close SSE connections | 2s | 17s |
| 5 | Stop RabbitMQ consumers + drain | 5s | 22s |
| 6 | Close Redis | 2s | 24s |
| 7 | Close HTTP server | 1s | 25s |
| 8 | Flush observability | 3s | 28s |
| 9 | Exit | ~0s | 28s |
| **Итого** | | | **28s (< 30s бюджет)** |

Запас 2s для непредвиденных задержек.

## 9.4 Force kill

Если graceful shutdown не завершился за `ORCH_SHUTDOWN_TIMEOUT` (30s):

1. Логирование `CRITICAL`: `"Graceful shutdown exceeded timeout, forcing exit"`.
2. `os.Exit(1)` — принудительное завершение.
3. Kubernetes / Docker перехватит и перезапустит контейнер.

## 9.5 Совместимость с rolling deployment

При rolling deployment (Kubernetes / Docker Compose):

1. Новый инстанс запускается и проходит readiness check.
2. Старому инстансу посылается `SIGTERM`.
3. Старый инстанс выполняет graceful shutdown (30s).
4. SSE-клиенты получают close event и переподключаются к новому инстансу.
5. Load balancer перенаправляет трафик на новый инстанс.

**Нулевой downtime** обеспечивается перекрытием readiness нового и shutdown старого инстанса.
