## 1. Общие сведения

### 1.1. Назначение документа

Настоящий документ описывает требования к модулю «Проверка договора» (далее — Модуль) в составе продукта «ContractPro». Документ предназначен для команды разработки (backend, frontend, ML), команды тестирования, UX/UI‑дизайнера, представителей заказчика (продукт, юридическая функция, бизнес‑заказчики), архитекторов и специалистов по интеграциям.

### 1.2. Область применения

Модуль предназначен для автоматизированного анализа договоров в юрисдикции РФ с целью выявления юридических и финансовых рисков, проверки наличия и корректности обязательных условий по ГК РФ и внутренней политике клиента, формирования рекомендаций по улучшению формулировок, а также предоставления краткого резюме для пользователей без юридического образования.

Модуль используется внутри продукта «ContractPro» через web‑интерфейс и API

## 2. Текущее состояние (как сейчас)

### 2.1. Текущее состояние продукта

На момент подготовки документа имеются: концепция продукта «ContractPro» и презентационные материалы; архитектура (Frontend / Backend / AI‑модуль / БД / облачная инфраструктура); определены ключевые пользовательские сценарии (создание договора, проверка договора, сравнение версий, краткое резюме).

## 3. Термины, определения и сокращения

**Договор —** юридически  документ, регулирующий отношения между сторонами.

**Тип договора —** разновидность договора, определяемая по характеру обязательств (например: услуги, поставка, подряд, аренда, NDA и др.).

**Пользователь —** авторизованное лицо, использующее систему (юрист, бизнес‑пользователь, администратор).

**Организация —** клиентская компания, работающая в системе «ContractPro».

**OCR —** оптическое распознавание символов (преобразование изображений текста в машиночитаемый текст).

**Уровень уверенности —** числовая оценка уверенности системы в классификации типа договора и/или извлечении параметров.

**Риск‑профиль —** cводная оценка договора на основе количества и критичности выявленных рисков и статусов обязательных условий.

## 4. Описание продукта и границы модуля

### 4.1. Область ответственности модуля

·      приём договора (файл/текст);

·      извлечение и структурирование текста;

·      определение типа договора и ключевых параметров;

·      проверку по чек‑листам обязательных условий и правилам выявления рисков;

·      формирование рекомендаций и альтернативных формулировок;

·      выдачу сводки/отчёта и (при необходимости) сравнение версий.

### 4.2. Вне рамок настоящего ТЗ

·      создание договора и редактирование как основной контур (описывается отдельным ТЗ);

·      управление жизненным циклом договора (подписание, исполнение, напоминания);

·      предоставление юридического заключения как услуги: модуль выдаёт рекомендации, окончательное решение принимает пользователь.

## 5. Роли пользователей и права доступа

**R‑1 Юрист:** Детальный просмотр результатов, экспорт отчётов, доступ к настройкам в пределах роли.

**R‑2 Бизнес‑пользователь:** Упрощённая сводка и ключевые риски «простым языком», экспорт доступен по политике организации.

**R‑3 Администратор организации:** Настройка политик проверки, чек‑листов, правил и прав доступа.

Доступ к договорам и результатам проверки должен быть ограничен рамками организации.

## 6. Пользовательские требования (UR)

**UR‑1. Загрузка договора.** Пользователь должен иметь возможность загрузить договор (файл или вставка текста) и получить статус обработки.

**UR‑2. Поддержка форматов.** Пользователь должен иметь возможность загружать договоры в форматах DOC/DOCX/PDF; поддержка изображений — при наличии.

**UR‑3. Автоопределение типа договора.** Система должна автоматически определять тип договора и учитывать его при анализе; при низкой уверенности запрашивать подтверждение.

**UR‑4. Список рисков.** Пользователь должен видеть список выявленных рисков с приоритизацией (высокий/средний/низкий).

**UR‑5. Пояснения по рискам.** Пользователь должен видеть по каждому риску краткое объяснение и ссылку на норму/политику (без дословного копирования текста закона).

**UR‑6. Рекомендации формулировок.** Пользователь должен получать рекомендуемые формулировки/правки для проблемных пунктов.

**UR‑7. Краткое резюме.** Бизнес‑пользователь должен видеть краткое резюме договора и ключевых рисков непрофессиональным языком.

**UR‑8. Сравнение с шаблоном/политикой.** Пользователь должен понимать, в каких пунктах договор отклоняется от стандартных условий компании (если политика/шаблон задан).

**UR‑9. Повторная проверка версий.** Пользователь должен иметь возможность повторно проверять доработанные версии договора и видеть изменения риск‑профиля.

**UR‑10. Выгрузка отчёта.** Пользователь должен иметь возможность выгрузить результаты проверки (например, PDF/DOCX) и/или получить защищённую ссылку.

**UR‑11. Обратная связь.** Пользователь должен иметь возможность отмечать полезность результата и оставлять комментарий (для улучшения качества).

**UR‑12. Настройка строгости.** Администратор должен иметь возможность настраивать строгость проверки и состав чек‑листов в пределах политики организации.

## 7. Функциональные требования (FR)

### 7.1. Загрузка и подготовка документа

FR‑1.1.1: Система должна принимать для проверки форматы: .doc, .docx, .pdf.

FR‑1.1.2: Дополнительно может поддерживаться загрузка изображений (.jpg, .png) для отдельных страниц.

FR‑1.2.1: Для скан‑PDF и изображений система должна выполнять OCR.

FR‑1.2.2: При частично нечитаемом тексте система должна выдавать предупреждение о возможной неполноте анализа и отмечать проблемные фрагменты.

FR‑1.3.1: Система должна выделять разделы, пункты, подпункты, приложения.

FR‑1.3.2: Система должна распознавать блок реквизитов сторон (при наличии).

## 7.2. Классификация договора и извлечение ключевых параметров

FR‑2.1.1: Система должна автоматически определять тип договора (услуги, поставка, подряд, аренда, NDA и др.).

FR‑2.1.2: Система должна сохранять уровень уверенности (оценку уверенности) результата определения.

FR‑2.1.3: При уровне уверенности ниже заданного порога система должна предложить пользователю подтвердить/выбрать тип договора.

FR‑2.2.1: Система должна извлекать ключевые параметры: стороны, предмет, цену/порядок расчетов, сроки, ответственность/неустойки, порядок приемки/качества, расторжение, применимое право/подсудность.

## 7.3. Проверка обязательных условий и рисков

FR‑3.1.1: Для каждого типа договора должен быть задан чек‑лист обязательных условий по ГК РФ и (при необходимости) судебной практике/политике организации.

FR‑3.1.2: Система должна проверять наличие и полноту указанных условий.

FR‑3.1.3: Статусы проверки: «условие найдено и корректно», «условие найдено, требуется внимание», «условие отсутствует/неопределённо».

FR‑3.2: Система должна выявлять риск‑конструкции, включая: одностороннее изменение условий; односторонний отказ; автоматическую пролонгацию; невыгодную подсудность/арбитраж; несимметричную ответственность и штрафы; неопределённые условия приемки/качества (перечень расширяем).

FR‑3.3.1: Для каждого риска присваивается уровень: высокий / средний / низкий.

FR‑3.3.2: Логика присвоения уровней риска должна быть настраиваемой администратором организации (R‑3).

FR‑3.4: По каждому риску должны отображаться пояснения и основания (норма/политика) без дословного копирования текста законодательства.

## 7.4. Рекомендации

FR‑4.1: Система должна предлагать рекомендуемые формулировки/правки для проблемных пунктов с учетом типа договора и роли стороны.

FR‑4.2: Если в продукте включено применение правок: пользователь может принять рекомендацию целиком или отредактировать; при применении создается новая версия договора и отмечаются изменения в риск‑профиле.

## 7.5. Результаты проверки и отчётность

FR‑5.1.1: Система формирует сводную оценку договора (низкий/средний/высокий риск) и количество рисков по уровням.

FR‑5.1.2: Система отображает краткое резюме договора (для R‑2).

FR‑5.2.1: Формируется детальный отчёт: риск, уровень, место в документе, пояснение, рекомендация.

FR‑5.2.2: Отчёт доступен в интерфейсе и для экспорта (PDF/DOCX) и/или по защищённой ссылке.

FR‑5.3.1: Система позволяет сравнивать две версии договора (до/после, наша/контрагента).

·FR‑5.3.2: Отображаются текстовые различия и изменения риск‑профиля.

## 7.6. Настройки и управление

FR‑6.1: Администратор организации должен иметь возможность настраивать политику проверки: включать/отключать пункты чек‑листов, настраивать строгость/правила.

FR‑6.2: Система должна обеспечивать разграничение прав доступа по ролям и организациям.

FR‑6.3: Система должна вести логирование: загрузка договоров, проверки, изменения настроек (см. раздел 10).

## 8. Нефункциональные требования (NFR)

### 8.1. Производительность и масштабируемость

NFR‑1.1: Время обработки стандартного договора объёмом до 30–40 страниц в текстовом формате (DOCX/PDF) — не более 60 секунд.

NFR‑1.2: Время обработки сканированного договора такого же объёма (с использованием OCR) — не более 120 секунд.

NFR‑1.3: Веб‑интерфейс (загрузка договора, отображение списка проверок, открытие результата) — ответ не более 2 секунд по p95.

NFR‑1.4: Архитектура Модуля должна поддерживать горизонтальное масштабирование компонентов API/веб без перерыва в работе сервиса.

### 8.2. Надёжность, доступность и восстановление

NFR‑2.1: Доступность сервиса (SLA) — не ниже 98,0% в месяц (если не задано иначе заказчиком).

NFR‑2.2: Операции записи критически важных данных (результаты проверки, настройки организаций, конфигурация правил) должны быть атомарными: только «успех» или «зафиксированная ошибка».

NFR‑2.3: RPO — не более 15 минут; RTO — не более 2 часов.

NFR‑2.4: Ключевые компоненты (API, воркеры анализа, БД, хранилище файлов) должны быть развернуты в отказоустойчивой конфигурации.

NFR‑2.5: При отказе отдельного экземпляра сервис должен сохранять работоспособность с возможной деградацией производительности, но без полной недоступности функционала.

### 8.3. Безопасность и конфиденциальность

NFR‑3.1: Передача данных должна осуществляться по защищённому каналу (TLS).

NFR‑3.2: Данные договоров и результатов проверки должны храниться в зашифрованном виде (в хранилище/БД).

NFR‑3.3: Доступ к данным должен быть ограничен рамками организации и ролью пользователя.

NFR‑3.4: Модуль должен вести журнал действий (авторизация, загрузка, проверки, просмотр результатов, изменения настроек/прав).

NFR‑3.5: Срок хранения логов безопасности и аудита определяется политикой ИБ продукта/клиента и должен поддерживаться системой.

### 8.4. Соответствие нормативным требованиям

NFR‑4.1: Обработка персональных данных должна соответствовать требованиям законодательства РФ о защите персональных данных (включая 152‑ФЗ) и внутренним политикам ИБ.

NFR‑4.2: В интерфейсе и пользовательской документации должны быть отражены юридические дисклеймеры: выводы Модуля носят рекомендательный характер и не заменяют юридическое заключение.

### 8.5. Удобство использования

NFR‑5.1: Сценарий «загрузить → запустить → получить результат» должен выполняться не более чем за 2–3 действия пользователя.

NFR‑5.2: Тексты ошибок и предупреждений должны быть на русском языке, без жаргона, с рекомендацией следующего шага.

### 8.6. Интеграции и совместимость

NFR‑6.1: Модуль должен корректно обрабатывать документы из распространённых систем (ЭДО, MS Office, Google Docs) без ручной доработки файлов пользователем.

NFR‑6.2: Для интеграции с внешними системами должен быть доступен REST API с аутентификацией, поддерживающий: передачу файла и метаданных, запуск проверки, получение статуса, получение результатов анализа.

NFR‑6.3: Форматы обмена — JSON/HTTP для метаданных и бинарная передача/ссылки для файлов; подробные схемы — в отдельной интеграционной спецификации.

### 8.7. Эксплуатация, мониторинг и сопровождение

Модуль должен поддерживать базовый мониторинг (ошибки, время обработки, нагрузка, очереди), оповещения и сбор метрик для эксплуатации.

## 9. Журналирование и аудит

Система должна журналировать: авторизацию, загрузку договоров, запуск проверок, просмотр результатов, изменения настроек и прав доступа, ошибки обработки. Записи должны содержать пользователя, время и тип операции.

## 10. Интеграции (API)

Для интеграции с внешними системами должен быть доступен REST API. Минимальные операции:

·      загрузка файла договора и метаданных;

·      запуск проверки;

·      получение статуса задачи;

·      получение результатов анализа.

Детальные схемы запросов/ответов и форматы хранимых файлов описываются в отдельной интеграционной спецификации.

## 11. Порядок контроля и приемки

Приемка модуля проводится по программе испытаний на контрольном наборе договоров и включает:

·      проверку извлечения текста и структуры для поддерживаемых форматов;

·      проверку определения типа договора и обработки низкой уверенности;

·      проверку чек‑листов обязательных условий по каждому типу договора;

·      проверку формирования сводки и детального отчёта, а также экспорта;

·      проверку выполнения требований по производительности
