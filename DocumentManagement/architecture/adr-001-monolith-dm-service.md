# ADR-001: Monolith DM Service

**Статус:** Принято

**Дата:** 2026-03-31

**Контекст:**

Document Management — stateful-домен, обрабатывающий и async-события (приём артефактов от DP/LIC/RE, request-response для semantic trees), и sync-запросы (CRUD документов/версий, чтение артефактов для оркестратора/UI). Необходимо выбрать deployment-модель.

---

## Рассмотренные варианты

### Вариант A: Monolith DM Service

Один Go-сервис, обрабатывающий и async events, и sync API. Внутри — чистое разделение по слоям (hexagonal).

**Плюсы:**
- Простота операционного управления (один deployment).
- Единая DB-транзакция для metadata.
- Единый connection pool к PostgreSQL, Redis, RabbitMQ.
- Соответствует подходу DP (один сервис = один домен).
- При нагрузке ~1000 договоров/сутки — более чем достаточно.

**Минусы:**
- Sync API и async consumer конкурируют за ресурсы.
- При масштабировании sync и async масштабируются вместе.

**Риски:**
- При резком росте нагрузки на API (например, batch-скачивание артефактов) может пострадать async processing.

**Совместимость с DP:** Полная. Единый стиль — один сервис на домен.

### Вариант B: Split API + Worker

Два отдельных deployment: DM API (sync) и DM Worker (async). Общая DB и Object Storage.

**Плюсы:**
- Независимое масштабирование sync и async.
- Изоляция нагрузки.

**Минусы:**
- Два deployment — сложнее операционно.
- Shared DB — нужна координация миграций.
- Больше конфигурации, мониторинга.
- Для ~1000 договоров/сутки — overengineering.

**Риски:**
- Конкурентный доступ к DB (row-level locking).

**Совместимость с DP:** Частичное отличие (DP — один сервис).

### Вариант C: CQRS (Command Query Responsibility Segregation)

Отдельные модели для записи (PostgreSQL) и чтения (read-optimized store, например Elasticsearch или materialized views).

**Плюсы:**
- Оптимальная производительность чтения при сложных запросах (поиск, фильтрация).
- Разделение нагрузки.

**Минусы:**
- Значительная архитектурная сложность.
- Eventual consistency между write и read моделями.
- Для текущих требований (простые CRUD-запросы) — избыточно.
- Дополнительная инфраструктура.

**Риски:**
- Stale reads. Сложность дебага.

**Совместимость с DP:** Не соответствует стилю. DP не использует CQRS.

---

## Решение: Вариант A — Monolith DM Service

### Обоснование

1. **Масштаб:** ~1000 договоров/сутки × ~10 артефактов на версию = ~10 000 записей артефактов/сутки. Один инстанс PostgreSQL + один Go-сервис легко справится.

2. **Согласованность с DP:** DP — один Go-сервис с ingress (async) + processing + egress. DM повторяет этот паттерн: ingress (async + sync) + application + egress. Единый стиль для всей системы.

3. **Операционная простота:** Один Docker-образ, один Helm chart, один набор healthcheck. При масштабе ContractPro v1 это оптимально.

4. **Эволюционность:** При росте нагрузки в 10–100× можно разделить на API + Worker (Вариант B) без изменения доменной модели и контрактов. Hexagonal архитектура позволяет это сделать заменой инфраструктурного слоя.

5. **Transactional outbox** решает проблему «DB commit + event publish» без распределённых транзакций.

6. **Idempotency на трёх уровнях** (Redis → DB unique constraints → S3 overwrite) обеспечивает надёжность при at-least-once delivery.

### Путь эволюции

При росте нагрузки: Monolith → Split API + Worker (Вариант B). Доменная модель, контракты и hexagonal-порты не меняются — только инфраструктурный слой.
